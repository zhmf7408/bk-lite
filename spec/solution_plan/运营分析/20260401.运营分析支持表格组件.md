# 运营分析支持表格组件 Solution Plan

日期：2026-04-01

## 1. 方案概览

### 1.1 目标

在不改变现有运营分析整体交互与权限体系的前提下，为仪表盘新增 `table` 组件能力，形成“创建 -> 查询 -> 展示 -> 保存 -> 复制复用”的闭环，满足记录级明细分析场景。

### 1.2 方案原则

- 严格以 Requirement 为实现基线，不扩展未提及功能。
- 轻量落地：优先复用现有组件框架与数据查询链路，减少新接口与新概念。
- 一致性优先：权限、状态反馈、保存与刷新行为与现有组件保持一致。
- 性能底线：分页必须走后端分页，避免全量拉取造成卡顿。

### 1.3 范围边界

#### In Scope

- 仪表盘新增 `table` 组件类型。
- 表格组件创建、编辑、复制、删除、保存回显。
- 组件级搜索（关键字、日期/时间范围）与重置。
- 列配置（默认列映射、顺序、显隐、组件内补列）。
- 后端分页查询（页码/每页条数/总数）。
- 多命名空间首列展示与命名空间筛选默认全选。
- 加载中、无数据、接口失败、无权限、处理失败状态反馈。

#### Out of Scope

- 导出（Excel/CSV）、复杂筛选编排、筛选方案保存。
- 复杂表格能力（透视、多级表头、汇总行、计算列）。
- 表格独立管理入口或页面化工作台。
- 因不同入参导致动态字段结构治理（本期按字段稳定假设处理）。

## 2. 关键方案决策（需求映射）

### 2.1 组件类型接入

- 在组件类型中新增 `table`。
- 组件创建、保存、编辑、删除、复制入口与现有组件一致。

### 2.2 默认列策略

- 决策：**默认列优先，探测列兜底**。
- 规则：
  - 数据源已配置默认列定义时，表格初始化优先采用默认列映射。
  - 未配置默认列定义时，按“当前入参查询结果第一条记录字段”生成初始列。
  - 探测字段不在默认列定义中时，展示名默认使用字段 `key`。
- 目标：首次创建后必须可见可用，不出现空白不可操作状态。

### 2.3 多命名空间规则

- 多命名空间场景首列固定为“命名空间名称”。
- 命名空间筛选默认全选，筛选项来源于后端现有查询能力返回的命名空间数据。
- 命名空间筛选参与正式查询，并与其他搜索条件统一生效。

### 2.4 查询与分页规则

- 搜索为组件级能力，搜索项由组件配置决定。
- 搜索执行后必须走后端重新请求，不做前端当前页本地过滤。
- 分页采用后端分页，返回至少包含：当前页数据、总数、当前页、每页条数。
- 翻页/刷新时保持当前搜索条件与列配置。

### 2.5 稳定性与权限规则

- 组件处理失败仅影响当前组件，不影响整页其他组件。
- 无权限时明确提示，不展示误导性空表。
- 表格组件的查看/新增/编辑/删除权限语义与现有仪表盘组件一致。

## 3. 分阶段落地计划

### 阶段 1：组件类型与基础链路（M1）

#### 目标

让 `table` 组件可创建、可绑定数据源、可保存、可回显。

#### 交付

- 组件类型列表接入 `table`。
- 组件配置模型支持 `table` 类型必要配置字段。
- 数据源能力判断中接入“支持表格类型”。
- 新增/编辑/保存/删除/复制流程打通。

#### 验收

- 用户可在仪表盘完成 `table` 组件新增、保存、编辑、复制、删除。
- 不改动现有 line/bar/pie/single 组件行为。

### 阶段 2：表格展示与列配置（M2）

#### 目标

实现“首屏可用”的列初始化与可维护列配置。

#### 交付

- 默认列优先 + 探测列兜底的初始化机制。
- 列顺序调整、列显隐、组件内补列能力。
- 多命名空间首列展示规则。
- 空值展示与长文本可读性保障。

#### 验收

- 首次创建后可见可用，不出现空白不可配置状态。
- 列配置保存后可回显，且仅作用于当前组件，不反写数据源默认列定义。

### 阶段 3：搜索与后端分页（M3）

#### 目标

保证大数据量下可用性与查询一致性。

#### 交付

- 组件级搜索（关键字、日期/时间范围）。
- 命名空间筛选接入并默认全选。
- 后端分页联动（页码、每页条数、总数）。
- 条件保持：翻页/刷新不丢失当前搜索条件。

#### 验收

- 搜索、翻页均触发后端查询。
- 分页结果结构完整且可回归。
- 大数据量场景无全量拉取导致的明显卡顿。

### 阶段 4：状态反馈与复用闭环（M4）

#### 目标

完善用户反馈与资产复用体验。

#### 交付

- 加载中、无数据、接口失败、无权限、处理失败状态覆盖。
- 复制仪表盘/组件时保留搜索配置、列配置、分页基础设置。
- 字段变化导致既有列不可用时给出明确提示。

#### 验收

- 异常态均有明确反馈，且整页不中断。
- 复制后配置可继续使用且行为一致。

## 4. 阶段验收清单（可执行）

### 4.1 功能验收

- 存在 `table` 组件类型并可完成全生命周期操作（增改删查、复制、保存回显）。
- 表格可展示记录级/对象级明细数据。
- 支持搜索、列配置、后端分页并可持续生效。

### 4.2 规则验收

- 默认列优先，缺省时按首条记录字段生成列。
- 多命名空间时首列命名空间，筛选默认全选并参与查询。
- 组件内补列仅影响当前组件。

### 4.3 权限与稳定性验收

- 权限语义与现有组件一致，不绕过组织与数据源权限。
- 失败场景只影响当前组件，不影响整页其他组件。

## 5. 依赖与协同

### 5.1 后端依赖

- 复用现有数据源查询能力，确保支持：
  - 统一查询口径（列探测与正式查询一致）。
  - 后端分页参数与返回结构。
  - 多命名空间结果与命名空间筛选项返回。

### 5.2 前后端协同要点

- 统一分页字段命名与语义（page/page_size/count/results）。
- 明确搜索参数协议（关键字、时间范围、命名空间筛选）。
- 明确无权限、失败、空数据状态码与前端展示规则。

## 6. 风险与应对

### 风险 1：PRD 与 Requirement 的组件类型口径不一致

- 现象：PRD 当前“可见图表类型”未显式包含 `table`，Requirement 已明确新增。
- 应对：本方案按 Requirement 落地；在需求链路中补充 PRD 同步动作，避免后续文档冲突。

### 风险 2：字段变化导致历史列配置失效

- 现象：数据源返回字段变化后，历史列配置可能出现缺失字段。
- 应对：组件提示不可用字段，并允许用户在组件内补列与重排。

### 风险 3：大数据量场景响应波动

- 现象：若误用全量拉取会导致卡顿或超时。
- 应对：强制后端分页；联调阶段用大数据量样本验证翻页与检索性能。

## 7. 回滚与降级策略

- 发布异常时可按组件类型开关回退到不展示 `table` 入口，不影响既有图表组件。
- 查询异常时按组件级降级展示错误态，不中断整页。
- 保留已保存仪表盘结构，避免因单组件问题导致资产不可读。
