# 系统管理-许可管理

日期：2026-04-07

## 背景

当前系统管理原型已提供“许可”设置页，用于统一管理许可码、模块授权状态、许可到期提醒，以及对象节点额度和日志容量等与授权容量相关的提醒能力。

从原型表达来看，许可管理承担的是平台级授权治理职责，不仅用于维护当前生效许可和历史许可，还用于向管理员清晰展示各平台模块的激活状态、默认免激活模块、容量使用情况和风险提醒状态。

现阶段需要将该原型整理为明确、可验证的 requirements 文档，统一本期许可管理能力的范围、需求项、验收口径与边界，避免后续设计、研发和验收对“许可管理是否只管许可码，还是同时覆盖模块授权和容量预警”产生理解偏差。

## 需求描述

系统管理新增“许可管理”需求，用于统一管理平台许可、模块授权和授权额度相关提醒，并支持管理员在系统管理 > 设置 > 许可 页面完成查看、配置和风险识别。

### 1）许可总览与许可列表

- 系统应在许可管理页展示当前生效许可列表。
- 许可列表中的每张许可应至少展示许可编号、有效期、包含模块和到期提醒信息。
- 系统应展示当前生效许可数量，帮助管理员快速了解当前许可规模。
- 系统应根据许可剩余有效期与当前提醒规则，区分展示正常状态与预警状态。
- 当许可进入提醒区间时，页面应以明确的预警样式提示风险，不应要求管理员自行计算是否临近到期。

### 2）默认提醒策略配置

- 系统应提供全局默认提醒配置能力，用于统一维护许可到期前的默认通知渠道、默认通知人员和默认提醒时间。
- 默认通知渠道应来源于系统管理的通知渠道能力，并支持多选。
- 默认通知人员应支持搜索、添加、移除和多选。
- 默认提醒时间应支持按“到期前 N 天”配置。
- 默认提醒时间修改后，应同步影响所有跟随默认策略的许可提醒展示与触发口径。
- 系统应明确提示默认提醒会通过所选通知渠道向所选通知人员发送提醒。
- 系统应向管理员明确说明默认提醒策略的作用范围，即对未单独配置的许可生效。

### 3）单许可提醒配置

- 系统应支持对单张许可配置独立提醒策略。
- 单许可提醒策略应至少支持“跟随默认”和“单独配置”两种模式。
- 当许可选择“跟随默认”时，应向管理员清晰展示当前继承的默认通知渠道、默认通知人员和默认提醒时间。
- 当许可选择“单独配置”时，应支持单独维护该许可的通知渠道、通知人员和提醒时间。
- 单许可通知渠道应支持多选，且来源于系统管理的通知渠道能力。
- 单许可通知人员应支持搜索、添加、移除和多选。
- 单许可配置确认后，许可卡片中的提醒文案应按最新口径回显。
- 单许可切回“跟随默认”后，应按默认策略生效，不再继续沿用此前的单独配置结果。

### 4）许可添加与激活

- 系统应提供“添加许可”入口，支持管理员录入新的许可码。
- 添加许可前，系统应向管理员展示当前系统注册码，便于管理员复制并用于申请许可。
- 系统应支持管理员执行注册码复制操作，并在复制成功或失败时给出明确反馈。
- 系统应支持管理员输入完整许可码并发起添加。
- 当许可码为空或不满足提交条件时，系统不应允许静默提交。
- 许可添加成功后，系统应给予明确成功反馈，并使新增许可进入许可管理视图。
- 新增许可完成后，系统应刷新许可列表。
- 模块授权状态和生效许可数量不要求在单次新增操作后实时联动刷新，可按页面刷新后的结果更新。

### 5）许可停用与历史许可查看

- 系统应支持对当前生效许可执行停用操作。
- 停用许可前，系统应给予明确的二次确认，提示停用后将限制相关模块继续新增授权资源。
- 停用后的许可不应继续计入生效许可数量。
- 停用许可后，已存在的授权节点不应受影响，但不应再允许继续新增节点或新增授权资源。
- 系统应提供历史许可查看入口，用于查看手动停用和已到期的历史许可。
- 历史许可中应至少展示许可编码、有效期、包含模块、停用时间和停用原因。
- 历史许可应作为独立视图存在，与当前生效许可列表区分展示，避免混淆当前状态。

### 6）平台模块授权状态展示

- 系统应在许可管理页展示平台模块清单，用于查看各模块授权状态。
- 平台模块区域应至少区分“免激活模块”“已激活模块”“未激活模块”三类状态。
- 系统应展示已激活模块数、免激活模块数和模块总数汇总信息。
- 免激活模块应以固定标识展示，不要求依赖许可激活。
- 已被许可覆盖的模块应展示为已激活状态；未覆盖的模块应展示为未激活状态。
- 平台模块授权展示应覆盖原型中的完整模块集合，而不是只展示当前已授权模块。

### 7）对象节点额度明细与提醒

- 对具备对象节点额度的模块，系统应支持查看对象节点额度明细。
- 对象节点额度明细应至少区分专用节点和通用节点两类额度池。
- 专用节点明细中应展示对象类型、已用额度、总额度、提醒阈值和当前状态。
- 当专用节点已用额度超过其专属额度时，系统应明确展示其已占用通用节点的情况。
- 通用节点明细中应展示通用节点的总使用情况，以及由哪些对象类型直接使用或因超额而占用。
- 节点状态至少应支持正常、预警、满额、超额等可区分状态。
- 系统应支持管理员重新统计节点额度，以刷新统计时间和当前展示结果。

### 8）节点提醒策略配置

- 系统应支持对对象节点额度配置提醒策略。
- 节点提醒策略应支持继承全局默认提醒的通知渠道与通知人员。
- 节点默认提醒配置应至少支持查看当前继承的默认通知渠道、默认通知人员和模块级默认提醒阈值。
- 默认提醒阈值应以“可用节点低于 N 个时提醒”的方式配置。
- 系统应支持为专用节点中的具体对象类型新增单独提醒配置。
- 每个专用节点类型在同一模块下只应允许存在一条单独提醒配置，避免重复配置。
- 专用节点单独提醒配置应至少包含节点类型、通知渠道、通知人员和提醒阈值。
- 专用节点单独提醒中的通知渠道应支持多选，通知人员应支持搜索、添加、移除和多选。
- 当专用节点未单独配置时，应默认跟随该模块的节点默认提醒策略。
- 系统应对节点提醒阈值进行输入校验，避免出现超过额度上限或无效值的结果。
- 节点提醒阈值单位应以“个”为准。

### 9）日志容量提醒

- 对日志中心模块，系统应支持查看日志存储容量使用情况。
- 系统应展示日志已用容量、总容量和剩余容量风险状态。
- 系统应支持管理员为日志容量配置独立提醒策略。
- 日志容量提醒策略应至少支持“跟随默认”和“单独配置”两种模式。
- 当日志容量提醒选择“跟随默认”时，应向管理员清晰展示当前继承的默认通知渠道、默认通知人员和默认提醒策略摘要。
- 当日志容量提醒选择“单独配置”时，应支持单独维护通知渠道、通知人员和“剩余低于 N GB 时提醒”的阈值配置。
- 日志容量通知渠道应来源于系统管理的通知渠道能力，并支持多选。
- 日志容量通知人员应支持搜索、添加、移除和多选。
- 日志容量提醒阈值应进行有效性校验，不应允许大于等于总容量的配置结果。
- 配置日志容量提醒成功后，系统应向管理员明确反馈当前通知渠道、通知人员和提醒阈值。
- 日志容量提醒阈值单位应以 GB 为准。

### 10）页面入口与交互一致性

- “许可”应作为系统管理下“设置”模块中的独立页签存在。
- 设置模块当前产品口径应至少包含“密钥”“许可”两个页签，并能清晰区分当前所在页签。
- 门户相关能力不属于本需求文档的约束范围。
- 许可管理页应至少包含“许可列表”“平台模块”“历史许可/提醒相关配置”三个层面的信息结构。
- 通知相关配置在页面中应形成一致的结构口径，至少体现“通知渠道”“通知人员”“提醒时间/提醒阈值”三个维度。
- 页面中的关键配置操作应支持“进入配置”“确认”“取消/关闭”的完整交互。
- 用户取消配置后，系统应恢复到取消前状态，不产生误生效结果。
- 关键操作失败时，系统应给予明确失败提示，不应让管理员误判为已生效。
- 许可管理页的操作方式应与系统管理其他设置页保持一致，不额外引入割裂的交互模式。

### 11）权限与可见范围

- 许可管理页的访问与操作权限应基于菜单权限控制。
- 仅内置超管角色应具备许可管理页的查看和操作权限。
- 普通用户不应看到许可管理页入口，也不应访问许可管理相关内容。

## 适用范围

- 系统管理 > 设置 > 许可 页面
- 平台许可列表与历史许可查看
- 平台模块授权状态展示
- 对象节点额度明细查看与提醒配置
- 日志中心容量提醒配置
- 许可管理菜单权限控制

## 非目标（本期不包含）

- 许可申请平台本身的外部申请流程与审批流能力
- 多租户、组织级或应用级差异化许可隔离模型
- 许可码生成算法、签名规则和解密机制等底层实现定义
- 自动续期、自动购买、自动扩容等商业化闭环能力
- 历史许可的恢复启用、版本对比或版本回滚能力
- 对平台模块描述文案、图标和排序规则提供自定义装修能力
- 除对象节点与日志容量外，将所有产品资源指标统一扩展为通用容量中心

## 验收标准

- 系统管理“设置”中存在“许可”页签，且可正常进入许可管理页面。
- 许可管理页可展示当前生效许可数量和生效许可列表。
- 每张生效许可至少可展示许可编号、有效期、包含模块和到期提醒信息。
- 管理员可展开默认提醒设置，并对默认通知渠道、默认通知人员和默认提醒时间进行配置。
- 默认通知渠道来源于系统管理通知渠道，且支持多选。
- 默认通知人员支持搜索、添加、移除和多选。
- 默认提醒时间支持按“到期前 N 天”配置，且修改后可影响跟随默认策略的许可展示结果。
- 管理员可对单张许可进入提醒配置，并在“跟随默认”“单独配置”两种模式间切换。
- 单许可单独配置时，可设置通知渠道、通知人员和提醒时间；确认后许可卡片文案可按最新配置回显。
- 页面存在“添加许可”入口，管理员可查看并复制系统注册码，并输入许可码发起添加。
- 许可码为空时系统可阻止提交并给出明确提示。
- 新增、修改、删除许可后，许可列表可刷新为最新结果。
- 管理员可对许可执行停用操作，且停用前存在明确确认提示；停用后仅限制继续新增节点或新增授权资源，不影响已存在节点使用。
- 页面存在历史许可入口，历史许可中至少可查看许可编码、有效期、模块、停用时间和停用原因，且历史许可仅包含手动停用和已到期许可。
- 平台模块区域可区分免激活模块、已激活模块和未激活模块，并展示对应数量汇总。
- 对具备对象节点额度的模块，管理员可进入对象节点额度明细视图。
- 对象节点额度明细可区分专用节点与通用节点，并展示各自的使用情况、阈值和状态。
- 节点提醒默认区可展示继承的默认通知渠道和默认通知人员，并支持配置模块级默认提醒阈值。
- 管理员可为专用节点类型新增单独提醒配置，单独配置中支持通知渠道多选、通知人员多选和阈值配置。
- 节点提醒配置中不允许重复选择同一专用节点类型，且阈值输入存在有效性校验。
- 日志中心可展示容量使用信息，并支持在“跟随默认”“单独配置”之间切换。
- 日志容量单独配置时支持通知渠道、通知人员和容量提醒阈值维护。
- 日志容量提醒阈值存在有效性校验，且保存成功后可给出明确反馈。
- 仅内置超管角色可查看和操作许可管理页，普通用户不可见。
- 页面中的关键配置操作在取消时不生效，在失败时有明确提示。

## 约束与边界

- Requirements 仅定义许可管理“为什么改、要改什么、验收什么”，不约束具体接口、存储结构、许可校验机制和实现方式。
- 本期需求以 `spec/prototype/系统管理/设置/许可.html` 为直接依据整理，若后续设计稿或产品结论发生变化，应以更新后的 requirements 或设计结论为准。
- 本期需求默认许可管理为平台级统一管理能力，不拆分不同租户、组织、应用之间的差异化授权模型。
- 本期需求默认许可管理由内置超管角色基于菜单权限维护，普通用户无权访问。
- 本期对象节点与日志容量提醒均视为许可管理页中的授权辅助能力，不代表已定义统一的跨产品容量中心方案。
