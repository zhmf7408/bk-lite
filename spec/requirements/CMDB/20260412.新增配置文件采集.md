# 新增配置文件采集

日期：2026-04-12

## 背景与问题

当前 CMDB 自动发现与采集能力可围绕主机和中间件实例采集基础属性，但缺少“操作系统层配置文件内容”的持续采集与版本查看能力。用户在排查配置漂移、核对线上配置变更、回溯问题发生时点时，无法直接在实例详情中查看某个配置文件的历史内容与差异，只能依赖人工登录主机排查，效率低且缺少统一留痕。

为提升主机配置可观测性，CMDB 需要新增“配置文件采集”能力，支持对主机上的文本配置文件进行周期采集、差异识别、版本沉淀与实例详情查看。

## 需求描述

CMDB 新增“配置文件采集”任务类型，用于采集操作系统层面的文本配置文件内容，并在实例详情中按文件维度查看版本历史与内容差异。

### 1）采集任务交互

- 配置文件采集任务的创建、编辑、查看、执行、状态反馈等交互方式参考现有主机采集任务。
- 用户在采集任务列表中可将配置文件采集任务与其他采集任务统一管理。
- 配置文件采集任务应支持与主机采集一致的基础信息录入方式，包括任务名称、目标范围、执行频率、状态查看与手动执行入口。
- 配置文件采集任务执行失败时，应展示明确失败原因，便于用户区分路径错误、权限不足、文件不存在、文件超限、文件非文本等场景。

### 2）采集任务录入项

- 配置文件采集任务新增“配置文件绝对路径”录入项，用户必须填写目标文件的绝对路径。
- 配置文件采集任务新增“配置文件名称”录入项。
- 当用户填写配置文件绝对路径后，系统默认使用路径中的文件名自动填充“配置文件名称”。
- 用户可在自动填充后手动修改“配置文件名称”，修改后的名称仅作为 CMDB 展示名称，不影响实际采集目标路径。
- 系统需兼容 Linux 与 Windows 的绝对路径格式，能够正确识别路径中的文件名称。
- 若配置文件绝对路径为空、格式非法或无法识别为绝对路径，系统应阻止保存并给出明确提示。

### 3）采集范围与边界

- 本期仅支持采集操作系统层面的文本文件内容。
- 本期仅支持以文件为粒度进行采集，不支持目录、通配符、文件夹批量扫描等模式。
- 本期仅支持读取当前主机上的目标文件内容，不支持通过 CMDB 页面直接编辑、回写或下发配置文件。
- 本期需同时考虑 Linux 与 Windows 主机场景。
- 若目标文件为二进制文件、压缩文件、图片、证书二进制内容或其他非文本内容，系统应拒绝作为有效采集结果入库，并给出明确提示。

### 4）文件大小限制

- 配置文件采集存在文件大小限制，系统需基于当前架构支持能力给出明确的单文件采集上限。
- 用户在创建或编辑采集任务时，应能够看到当前支持的文件大小限制说明。
- 当目标文件超过系统支持的大小上限时，本次采集不得生成新版本，并应在任务结果中明确提示“文件超限”及对应限制信息。
- 文件大小限制应以“可验证、可配置或可明确说明”的方式对外呈现，不允许仅在执行失败后给出笼统错误信息。

### 5）内容存储位置

- 配置文件采集生成的文件内容与版本正文，不写入 CMDB 实例属性中。
- 配置文件采集生成的文件内容与版本正文统一存放在 MinIO 中。
- CMDB 中仅保留与配置文件采集相关的必要元数据，用于任务管理、文件列表展示、版本列表展示与内容定位。
- 版本内容的存储方式不得影响实例详情页的基础属性展示性能。

### 6）采集频率与执行规则

- 配置文件采集任务默认采集频率为 10 分钟。
- 用户在新建任务时，系统默认带出 10 分钟采集频率。
- 配置文件采集任务应支持周期执行，并可查看最近执行结果。
- 每次采集时，系统读取目标文件当前内容，并与该文件上一个已保存版本的内容进行对比。
- 当本次采集内容与上一版本一致时，系统不生成新版本，但应记录本次采集已完成。
- 当本次采集内容与上一版本存在差异时，系统生成新的文件版本。
- 新生成的版本号使用采集时间戳。

### 7）版本管理与内容查看

- 配置文件版本以“单个实例下的单个配置文件”为单位进行管理。
- 同一实例下可维护多个配置文件，每个配置文件拥有独立的版本历史。
- 用户查看某个配置文件时，可看到该文件的版本列表。
- 版本列表至少展示版本号、采集时间、采集结果状态等关键信息。
- 用户点击某个版本后，可查看该版本的完整文本内容。
- 本期支持两个版本之间的文本差异对比，便于用户识别配置变化。
- 系统应能够基于版本元数据定位 MinIO 中存储的对应版本内容。

### 8）实例详情查看入口

- 配置文件应在主机实例详情的专用 Tab 中查看。
- 该 Tab 以列表方式展示当前实例下已纳入采集的多个配置文件名称。
- 用户点击某个配置文件名称后，可进入该文件的版本列表。
- 用户在版本列表中点击具体版本后，可查看版本内容；选择两个版本时，可查看文本差异。
- 当实例下暂无配置文件采集记录时，Tab 中应展示空状态提示，而非报错信息。

### 9）展示与命名规则

- 配置文件列表默认展示“配置文件名称”，同时应能让用户区分其对应的实际采集路径。
- 当配置文件名称被用户手动修改后，实例详情中优先展示用户维护后的名称。
- 同一实例下允许存在多个配置文件采集项，用户应能清晰区分不同文件。
- 版本号展示应与时间戳规则保持一致，避免同一页面中出现多种版本号样式。

## 适用范围

- CMDB：自动发现与采集任务中新增加“配置文件采集”任务类型。
- CMDB：主机实例详情新增配置文件查看专用 Tab。
- CMDB：配置文件版本列表、版本内容查看与版本差异对比。
- CMDB：Linux 与 Windows 主机上的操作系统层文本配置文件采集。
- CMDB：配置文件版本正文存放于 MinIO，CMDB 仅管理元数据与展示链路。

## 约束与边界

### In Scope

- 配置文件采集任务的创建、编辑、执行与结果查看。
- 操作系统层文本配置文件的周期采集。
- 基于内容差异的版本生成。
- 实例详情中的配置文件列表、版本列表、版本内容与文本差异查看。
- 配置文件版本正文存储到 MinIO，CMDB 实例属性中不承载版本正文。

### Out of Scope

- 二进制文件采集与解析。
- 目录级批量采集、通配符扫描、自动发现整机全部配置文件。
- 配置文件在线编辑、回写主机、版本回滚。
- 跨实例的配置文件聚合分析与统一对比。
- 非主机场景下的专用展示入口建设。

## 验收标准

- 用户可在 CMDB 中创建配置文件采集任务，且其交互方式与主机采集任务保持一致。
- 配置文件采集任务录入时必须包含“配置文件绝对路径”和“配置文件名称”两个字段。
- 当用户填写配置文件绝对路径后，系统会默认带出路径中的文件名作为配置文件名称，且允许用户手动修改。
- 系统可正确识别 Linux 与 Windows 的绝对路径，并正确提取默认文件名。
- 当路径非法或非绝对路径时，系统阻止任务保存并给出明确提示。
- 系统仅采集操作系统层文本文件；当目标文件为非文本内容时，系统明确提示且不生成有效采集结果。
- 用户可明确获知当前支持的单文件采集大小上限；当文件超限时，任务结果明确提示超限原因，且不生成新版本。
- 配置文件版本正文不会写入 CMDB 实例属性中，而是统一存放于 MinIO；CMDB 中仅保留支撑展示与检索所需的元数据。
- 新建配置文件采集任务时，默认采集频率为 10 分钟。
- 每次采集后，系统会将本次内容与上一版本进行对比；内容无差异时不生成新版本，有差异时生成新版本。
- 新版本的版本号为时间戳，且在版本列表中可被稳定展示。
- 用户可在主机实例详情的专用 Tab 中查看该实例下的多个配置文件。
- 用户点击配置文件后可查看版本列表，点击具体版本后可查看版本内容。
- 用户可选择两个版本进行文本差异对比。
- 当实例下没有配置文件采集记录时，页面显示空状态，不影响实例详情其他信息展示。
